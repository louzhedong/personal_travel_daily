# 未来 Roadmap / TODO

这份文档用于约束后续功能迭代方向。默认以后新增功能优先从这里选题，避免需求发散。

## 当前产品基线

截至当前版本，项目已经具备这些核心能力：

- 地图选区与旅行记录录入
- 多旅伴切换与权限边界
- 旅行记录详情、编辑与图片查看
- 攻略搜索、收藏、关联与回跳
- 行程时间线与地图联动
- 本地数据备份、导入与恢复
- IndexedDB 持久化与本地攻略缓存

这意味着下一阶段不适合再重复补基础 CRUD，而更适合补“结构化回看、可检索性、可运营性和长期可维护性”。

## 优先级说明

- `P1`: 建议优先做，能明显放大现有数据价值
- `P2`: 值得做，但依赖前面能力收口后推进更稳
- `P3`: 中长期增强项，适合后续版本继续演进

## 最想做的

### 0. 数据库升级到 MySQL

- 优先级：`P0`
- 为什么最值得做：当前项目虽然已经有 `IndexedDB`、本地缓存和导入导出能力，但核心数据仍然是浏览器本地优先。这会限制后续很多真正有长期价值的能力，比如账号体系、跨设备同步、多人协作、服务端搜索、服务端统计和分享页发布。把数据底座升级到 MySQL，会是后续功能真正放开的前提。
- 建议范围：
  - 设计服务端数据模型，覆盖用户、旅行记录、已保存攻略、搜索历史、行程实体等核心对象
  - 新增服务端持久化层，替代当前仅浏览器本地持久化的主路径
  - 设计本地 `TravelStore` 到 MySQL 的迁移和同步策略
  - 明确主键、去重键、外键关系和软删除 / 冲突处理策略
  - 保留本地导出作为兜底能力
- 预期影响：
  - 为“云端同步与账号体系”铺路
  - 为“分享页 / 旅行故事页”提供稳定发布数据源
  - 为后续统计、搜索、推荐和协作功能提供统一数据库基础

## TODO Top 10

### 1. 行程集合 / Trip Collection

- 优先级：`P1`
- 为什么值得做：现在记录是“散点”，时间线已经能回看，但还缺少“这次去新疆”“2025 日本春游”这种更高层的行程组织能力。
- 建议范围：
  - 新增 `Trip` 概念
  - 一组旅行记录可归属到同一个行程
  - 行程支持名称、封面、起止日期、备注
  - 时间线支持按行程维度聚合查看

### 2. 旅行记录全文搜索与筛选

- 优先级：`P1`
- 为什么值得做：目前记录主要靠地图、时间线和列表浏览，找旧内容成本仍然高。
- 建议范围：
  - 支持按地区、城市、描述关键词搜索
  - 支持按旅伴、年份、国内/国际组合筛选
  - 支持从搜索结果直接打开详情或定位地图

### 3. 行程统计中心

- 优先级：`P1`
- 为什么值得做：现在已有基础 `StatsPanel`，但还没有把“记录”转成更有成就感和回顾价值的数据视图。
- 建议范围：
  - 总旅行天数、城市数、地区数、国家数
  - 年度统计与月度分布
  - 最常访问地区 / 最活跃旅伴
  - 可作为首页第二屏或独立面板

### 4. 记录标签系统

- 优先级：`P1`
- 为什么值得做：标签会显著提升搜索、筛选和内容组织能力，也能为后续 AI 或推荐能力打基础。
- 建议范围：
  - 给旅行记录添加标签
  - 系统预置标签与自定义标签并存
  - 支持按标签筛选时间线、列表和统计

### 5. 攻略卡片沉淀与行前清单

- 优先级：`P2`
- 为什么值得做：攻略搜索已经有了，但现在更多是“查过、存过”，还没有转成真正可执行的旅行准备内容。
- 建议范围：
  - 从收藏攻略中提取“必去点 / 注意事项 / 交通提示”
  - 支持用户勾选或手动整理成清单
  - 清单可绑定到某个行程

### 6. 地图回放模式

- 优先级：`P2`
- 为什么值得做：现有地图、轨迹、时间线已经具备基础，差一步就能形成非常强的回看体验。
- 建议范围：
  - 按时间顺序播放旅行记录
  - 地图依次高亮地区与轨迹
  - 可切换播放速度
  - 与时间线保持同步高亮

### 7. 分享页 / 旅行故事页

- 优先级：`P2`
- 为什么值得做：当前数据主要停留在“自己看”，分享能力会明显提升产品表达力。
- 建议范围：
  - 将单次行程或年度回顾生成可阅读页面
  - 展示封面、时间线、照片和关联攻略
  - 第一阶段可先做本地只读页，不必急着上真正公网分享

### 8. 云端同步与账号体系

- 优先级：`P2`
- 为什么值得做：现在虽然有备份恢复，但仍是纯本地模式。只要用户开始长期使用，就会自然需要跨设备同步。
- 建议范围：
  - 账号登录
  - 云端同步 TravelStore
  - 冲突处理与最近修改优先策略
  - 保留本地导出作为兜底

### 9. 攻略质量治理与来源管理

- 优先级：`P3`
- 为什么值得做：攻略搜索已经可用，但随着适配器和数据量增加，结果质量、来源可信度和重复内容会成为问题。
- 建议范围：
  - 来源优先级配置
  - 相似攻略合并或去重
  - 结果质量打分
  - 失效来源与抓取失败监控

### 10. 架构层继续拆分与状态测试补强

- 优先级：`P3`
- 为什么值得做：前端已经完成第一轮瘦身，但如果后续继续上新功能，不继续收口会再次回到 `App.tsx` 膨胀。
- 建议范围：
  - 抽离 `useGuideSearchState`
  - 抽离 `useMarkerDetailState`
  - 增加 hook 层单测
  - 为关键联动流程补集成测试

## 推荐执行顺序

建议按下面顺序推进：

1. 数据库升级到 MySQL
2. 行程集合 / Trip Collection
3. 旅行记录全文搜索与筛选
4. 行程统计中心
5. 记录标签系统
6. 攻略卡片沉淀与行前清单
7. 地图回放模式
8. 分享页 / 旅行故事页
9. 云端同步与账号体系
10. 攻略质量治理与来源管理
11. 架构层继续拆分与状态测试补强

## 选题原则

以后如果要从这份清单中挑一个功能开做，优先满足这三个条件：

1. 能复用已有地图、时间线、攻略或存储能力
2. 能显著提升“回看价值”或“整理效率”
3. 不会把业务规则重新塞回页面容器层
